\chapter{系统全貌}

\section{系统由哪两部分组成}

\subsection{Windows 侧：记忆内核}

Windows 侧位于 \texttt{D:\textbackslash openclaw}，它不是 Gateway 本体，而是一个轻量 memory kernel。它负责：

\begin{itemize}
  \item working memory
  \item summary / reflection / selfcheck
  \item knowledge partition
  \item runtime snapshot
  \item 规则化清理与压缩
\end{itemize}

\subsection{WSL 侧：真实 OpenClaw 运行时}

WSL 侧位于 \texttt{/home/park/openclaw}，它负责：

\begin{itemize}
  \item CLI 入口
  \item Gateway 服务
  \item chat / agent 方法分发
  \item auto-reply / agent turn 执行链
  \item 通过 sidecar bridge 调用 Windows 侧记忆内核
\end{itemize}

\section{为什么 \texttt{D:\textbackslash openclaw} 是系统的窝}

这是当前系统的\textbf{约定根目录}。理由如下：

\begin{enumerate}
  \item Windows 侧 memory kernel 全部在这里
  \item 启动脚本会把它映射成 WSL 可见的 \texttt{/mnt/d/openclaw}
  \item WSL 侧 sidecar bridge 默认也指向这个路径
  \item memory 里的 \texttt{working.json} 和 \texttt{meta.json} 都已经把它记成 canonical home
\end{enumerate}

\section{真实运行链}

当前真实运行链如下：

\begin{enumerate}
  \item 用户启动 \texttt{Start-OpenClaw-WSL-Interactive.cmd}
  \item 脚本进入 WSL，并导出 \texttt{OPENCLAW\_EXTERNAL\_MEMORY\_ROOT=/mnt/d/openclaw}
  \item WSL 调用 \texttt{/home/park/.local/bin/openclaw gateway start}
  \item Gateway 服务运行
  \item 用户发起 \texttt{chat.send} 或相关 agent 流程
  \item OpenClaw 完成一次真实聊天或 agent turn
  \item sidecar bridge 触发 \texttt{D:\textbackslash openclaw\textbackslash main.py}
  \item Windows 侧 memory 执行一轮 compact cycle
  \item \texttt{working.json}、\texttt{runtime.json} 等状态更新
\end{enumerate}

\section{本次真实验收结果}

本手册生成前已经做过一次真实聊天主路径验收：

\begin{itemize}
  \item 使用真实 \texttt{chat.send}
  \item Gateway 返回 \texttt{status = started}
  \item 随后检查 \texttt{working.json}
  \item \texttt{step\_count} 从 42 增加到 43
  \item \texttt{runtime.json} 新增了 \texttt{step = 43} 的记录
\end{itemize}

这说明当前不是“只有单独运行 \texttt{main.py} 才有效”，而是\textbf{真实聊天路径已经带动了这套 memory}。

\chapter{逐文件解释}

\section{Windows 侧解释}

\subsection{\texttt{AGENTS.md}}
这是当前 OpenClaw Memory Sidecar 的操作契约。它不是算法代码，但它定义了这套系统该如何理解任务、如何控制行为边界、如何避免无关扩张。

\subsection{\texttt{main.py}}
这是 OpenClaw Memory Sidecar 的总入口。任何一次 memory cycle 都从这里开始。它负责调用各个 core 模块，并决定本轮是否进行检索、反思、summary 压缩、自检、runtime 记录。

\subsection{\texttt{Start-OpenClaw-WSL-Interactive.cmd}}
这不是 Gateway 主实现，但它是用户最直观的启动入口。当前它有一个关键职责：把 \texttt{/mnt/d/openclaw} 作为 external memory root 显式导出给 WSL 运行时。

\subsection{\texttt{core} 目录}
\texttt{core} 目录是 Windows 侧记忆内核的逻辑层。这里的每个文件都尽量保持单一职责，避免把所有逻辑压进 \texttt{main.py}。

\subsection{\texttt{memory} 目录}
\texttt{memory} 目录是持久化状态层。它既不是原始日志仓库，也不是 transcript 仓库，而是压缩后的长期状态与知识。

\section{WSL 侧解释}

\subsection{\texttt{bin/openclaw}}
这是 WSL 里用户最终调用到的 \texttt{openclaw} 命令。它本身不是核心逻辑，只是把命令转发给 Node CLI 入口。

\subsection{\texttt{package.json}}
这里定义了 \texttt{openclaw} 作为 CLI 包如何暴露给系统，是 CLI 身份和构建脚本的根。

\subsection{\texttt{src/cli/gateway-cli/run.ts}}
这是前台运行 Gateway 的命令逻辑。它负责解析运行参数，并把启动动作交给真正的 server 启动函数。

\subsection{\texttt{src/cli/gateway-cli/run-loop.ts}}
这是 Gateway 服务的生命周期主循环。它负责服务启动、关停、重启、锁管理、信号处理、重启后的继续运行。

\subsection{\texttt{src/infra/external-memory-kernel.ts}}
这是本次集成的核心 bridge。它不直接实现 memory，而是作为 WSL 侧的 sidecar 调用器，去触发 \texttt{D:\textbackslash openclaw} 中的 OpenClaw Memory Sidecar。

\subsection{\texttt{src/gateway/server-methods/chat.ts}}
这是最重要的真实用户主路径之一。用户发一句话，本质上会走到这里。现在这个文件在聊天成功或失败后会触发一次 sidecar memory run。

\subsection{\texttt{src/gateway/server-methods/agent.ts}}
当系统通过 RPC 形式直接触发 agent 流程时，会走这个文件。它现在也会在完成后触发 memory sidecar。

\subsection{\texttt{src/auto-reply/reply/agent-runner.ts}}
这是 reply-agent 的实际执行主路径。对于真实 agent turn，这是最关键的完成点之一。它现在在非 heartbeat 情况下会触发 memory sidecar。

\chapter{如何维修}

\section{最常见问题}

\subsection{发了消息，但 memory 不更新}

优先按下面顺序查：

\begin{enumerate}
  \item \texttt{python main.py} 是否能独立运行
  \item WSL 源码改完后是否重新 \texttt{pnpm build}
  \item Gateway 服务是否真的重启到新构建
  \item \texttt{chat.send} 实际是否走到当前接入路径
  \item \texttt{D:\textbackslash openclaw} 是否仍存在、是否仍包含 \texttt{main.py}
\end{enumerate}

\subsection{Gateway 服务起不来}

\begin{enumerate}
  \item 查 \texttt{journalctl --user -u openclaw-gateway.service -n 200 --no-pager}
  \item 查 \texttt{/tmp/openclaw/openclaw-YYYY-MM-DD.log}
  \item 判断是服务没起来、起来后秒退、还是 UI 构建卡住
\end{enumerate}

\subsection{系统记不住“窝”}

查三处：

\begin{enumerate}
  \item \texttt{Start-OpenClaw-WSL-Interactive.cmd} 是否仍导出 \texttt{OPENCLAW\_EXTERNAL\_MEMORY\_ROOT=/mnt/d/openclaw}
  \item \texttt{src/infra/external-memory-kernel.ts} 默认 root 是否仍是 \texttt{/mnt/d/openclaw}
  \item \texttt{memory/meta.json} 与 \texttt{memory/working.json} 是否仍把 \texttt{D:\textbackslash openclaw} 视为 canonical home
\end{enumerate}

\section{推荐维修流程}

\begin{enumerate}
  \item 先备份
  \item 只改一个挂点或一个模块
  \item 跑类型检查
  \item 重新构建 WSL 侧 dist
  \item 重启 gateway
  \item 发一条真实消息
  \item 检查 \texttt{working.json} 的 \texttt{step\_count} 是否增长
  \item 检查 \texttt{runtime.json} 是否新增记录
\end{enumerate}

\chapter{最后说明}

这份手册的设计目的不是“把所有文件机械堆一起”，而是让后来维修的人能回答四个问题：

\begin{enumerate}
  \item 系统现在到底怎么跑
  \item 哪些文件是真正关键的
  \item 真实消息为什么会更新 memory
  \item 出故障时应该先看哪里、先修哪里
\end{enumerate}

因此，这份文档同时保留了：

\begin{itemize}
  \item 结构解释
  \item 运行链解释
  \item 维修流程
  \item 完整源码与状态快照
\end{itemize}

如果后续继续演化系统，建议每次大改后重新导出一版手册，以避免“代码已经变了，手册还是旧的”这种二次故障。
