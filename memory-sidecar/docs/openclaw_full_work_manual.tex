\chapter{System Summary}
The current sidecar design is intentionally split into two layers:
\begin{itemize}
  \item \textbf{memory-sidecar}: controller, observability, recovery, runtime state snapshots.
  \item \textbf{memory-lancedb-pro}: canonical long-term memory engine.
\end{itemize}
The old state where sidecar also behaved like a competing retrieval engine is no longer the intended architecture.
\chapter{Scope}
This manual is not a full OpenClaw platform manual. It covers only the sidecar-relevant surface:
\begin{itemize}
  \item \texttt{memory-sidecar/main.py} and \texttt{memory-sidecar/core/**}
  \item the runtime-side durable emitter in \texttt{src/infra/external-memory-kernel.ts}
  \item the root resolver in \texttt{src/infra/external-memory-root.ts}
  \item the bridge in \texttt{src/infra/memory-lancedb-pro-bridge.ts}
  \item the plugin backend in \texttt{extensions/memory-lancedb-pro/}
\end{itemize}
Other OpenClaw modules matter only where they trigger sidecar events or provide the long-term memory backend.
\chapter{Architecture Boundary}
\section{What Sidecar Owns}
\begin{itemize}
  \item event durability
  \item commit, ack, replay
  \item mode switching
  \item health, pressure, noise, complexity tracking
  \item reflection and self-check triggering
  \item runtime traceability
  \item operational snapshots such as working, summary, strategy, reflection, selfcheck, and runtime
\end{itemize}
\section{What memory-lancedb-pro Owns}
\begin{itemize}
  \item long-term recall
  \item long-term store, update, forget
  \item hybrid retrieval
  \item rerank
  \item scope isolation
  \item lifecycle and decay on the long-term memory side
\end{itemize}
\section{Explicit Non-Goals For Sidecar}
Sidecar should not rebuild a second long-term memory engine when \texttt{memory-lancedb-pro} is available.

\chapter{Current Sidecar Runtime Model}
\section{Execution Modes}
The current sidecar has four operational entry modes:
\begin{itemize}
  \item \texttt{python3 main.py}: run one control cycle directly
  \item \texttt{python3 main.py --once}: claim one queued event and process it
  \item \texttt{python3 main.py --worker}: poll the file queue continuously
  \item \texttt{python3 main.py --queue-status}: print queue and root status
\end{itemize}
\section{Real Runtime Boundary}
The request-side OpenClaw runtime does not execute the full Python cycle inline anymore. It only persists a durable event through \texttt{src/infra/external-memory-kernel.ts}. The sidecar worker later claims that event and runs \texttt{core/cycle.py}. This is the key boundary between request-time work and post-request sidecar work.
\section{Single-Cycle Internals}
Within one sidecar cycle, the current flow is:
\begin{enumerate}
  \item resolve active event context
  \item load operational snapshots
  \item resolve canonical backend or JSON fallback
  \item build runtime signals
  \item select mode and recall plan
  \item execute one control step
  \item optionally reflect, self-check, and compact local operational state
  \item append runtime record, trace, acknowledgement, and commit metadata
\end{enumerate}

\chapter{Root Resolution}
\begin{enumerate}
  \item Canonical runtime root: \texttt{/home/park/openclaw}
  \item Canonical memory root: \texttt{/home/park/openclaw/memory-sidecar}
  \item Runtime-side resolution order: \texttt{OPENCLAW\_EXTERNAL\_MEMORY\_ROOT} -> \texttt{OPENCLAW\_RUNTIME\_ROOT + /memory-sidecar} -> code default
  \item Current Python sidecar code still uses \texttt{OPENCLAW\_EXTERNAL\_MEMORY\_ROOT} first and then falls back to the package parent path. Treat the explicit environment variable as authoritative until both sides are fully unified.
  \item Historical paths such as \texttt{D:\textbackslash openclaw} and \texttt{/mnt/d/openclaw} are deprecated only
\end{enumerate}
\chapter{Transactional Flow}
\begin{enumerate}
  \item A runtime trigger persists an event before doing any sidecar work.
  \item Request-side runtime stops after durable emit; it does not run a full memory cycle inline.
  \item A worker later claims one queued or replayable event.
  \item Sidecar checks whether that event was already acknowledged or already applied.
  \item If already applied, sidecar performs an idempotent duplicate no-op and writes the proper acknowledgement.
  \item If not yet applied, sidecar runs one signal-driven control cycle and then writes acknowledgement and commit records.
\end{enumerate}
\chapter{Replay And Recovery}
Failed and replayable events remain durable. The next worker pass may replay them. If effects already happened but ack was missing, sidecar detects that by event id and runtime state, then finalizes with a duplicate no-op acknowledgement instead of repeating memory side effects.
\chapter{Signal-Driven Orchestration}
The old rigid loop used fixed partition selection, fixed top-k, and fixed cadence for reflection and self-check. The current controller is signal-driven.
\section{Representative Signals}
\begin{itemize}
  \item repeated\_steps
  \item contradiction
  \item consecutive\_failures
  \item rapid\_health\_drop
  \item interruption\_recovery
  \item low\_health
  \item high\_pressure
  \item high\_noise
  \item abnormal\_growth
  \item low\_yield\_recall
\end{itemize}
\section{Controller Decisions}
The sidecar decides when to recall, why to recall, whether to store or block a write, when to reflect, when to run self-check, when to compact local operational state, and when to switch between normal, stability, and convergence.
\chapter{Traceability}
Every run should emit an append-only JSONL trace. The trace is expected to include at least:
\begin{itemize}
  \item event id
  \item request id
  \item trigger source
  \item replay flag
  \item replay attempt
  \item whether recall was requested
  \item recall reason
  \item backend used
  \item recall query
  \item returned memory count
  \item selected memory ids
  \item discard reasons
  \item health, pressure, noise, complexity
  \item current mode
  \item selected strategy
  \item reflection, self-check, and cleanup decisions
  \item whether memory write was allowed
  \item patch proposal and patch apply outcome
  \item runtime step
  \item ack id
  \item final outcome and failure reason
\end{itemize}

\chapter{Patch Safety}
Patch proposal may remain available, but automatic patch apply must remain off by default. Apply is only allowed behind explicit configuration. Proposal, apply, and failure must all be traced. Failure must not be silent.

\chapter{Validation Status}
As of 2026-03-23, the implementation has been validated with Python unit tests, TypeScript unit tests for the launcher, direct sidecar execution, and launcher-driven smoke tests. The durable event flow, replay path, canonical long-term backend routing, and signal-driven orchestration all passed local validation.
\chapter{Sidecar-Only Review Notes}
From a sidecar maintenance perspective, the current strongest properties are:
\begin{itemize}
  \item request-side emit and worker-side consume are now separated
  \item durable files are append-only where possible
  \item long-term memory is delegated to \texttt{memory-lancedb-pro}
  \item local JSON files remain readable operational snapshots
\end{itemize}
The main remaining caveat is root-resolution convergence: runtime-side TypeScript and Python sidecar logic are close, but not yet perfectly identical when only \texttt{OPENCLAW\_RUNTIME\_ROOT} is set. For stable operation, set \texttt{OPENCLAW\_EXTERNAL\_MEMORY\_ROOT} explicitly.
\chapter{Migration Notes}
\begin{itemize}
  \item \texttt{knowledge/*.json} is no longer the sole source of truth for long-term memory.
  \item Local JSON snapshots remain, but only as operational state.
  \item Durable event, ack, commit, trace, and recovery files are now part of normal operation.
  \item Fixed-step orchestration has been replaced by signal-driven orchestration.
\end{itemize}

\chapter{Tradeoffs And Deferred Items}
\section{Main Tradeoffs}
\begin{itemize}
  \item thin bridge instead of a second memory engine in Python
  \item child-process model instead of a heavyweight daemon
  \item JSON fallback retained for compatibility but not treated as canonical when lancedb-pro is present
  \item append-only JSONL traces for readability and auditability
\end{itemize}
\section{Deliberately Deferred}
\begin{itemize}
  \item richer selected-memory and discard-reason explanations
  \item more advanced replay prioritization
  \item dedicated trace dashboard
  \item broader patch allowlist governance
\end{itemize}
